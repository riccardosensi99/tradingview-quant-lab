# Registry Integration

Implemented in `scripts/generation/registry-entry-from-idea.ts` (`buildExperimentalEntry`, `commitApprovedIdeas`), reusing `scripts/research/registry-io.ts`'s `writeRegistry`/`upsertStrategy` unmodified. Tested in `tests/registry-entry-from-idea.test.ts`, including an explicit "no write before approval" test using a tmpdir. Background: [docs/decisions/0003-idea-stage-registry-entries.md](../../../docs/decisions/0003-idea-stage-registry-entries.md).

## The approval gate is structural, not just documented

`buildExperimentalEntry` and `commitApprovedIdeas` never import `node:fs`, `scripts/research/registry-io.ts`'s `writeRegistry`, or any MCP tool — they only build and fold `StrategyRegistryEntry` objects in memory. The only function that actually touches disk is `writeRegistry`, and [SKILL.md](SKILL.md)'s process step 6 is a hard stop before it's ever called: the agent must present the generation report and receive the human operator's explicit approval before proceeding to step 7. This mirrors [tradingview-strategy-research](../tradingview-strategy-research/SKILL.md)'s own confirm-before-first-mutating-call pattern, just enforced earlier and harder — there is no code path in `scripts/generation/` that writes to the registry on its own.

## What gets written for an approved idea

`buildExperimentalEntry(idea, reportPath, date)` produces a `StrategyRegistryEntry` with:

- `status: "experimental"`, `stage: "idea"` — `pine_script_id` and `metrics` are **absent**, not fabricated as placeholders or zeros (`scripts/schemas/registry.ts`'s `superRefine` only allows this at `stage: "idea"` — see the ADR)
- `id`, `name`, `symbol_universe` (from `initial_symbols`), `timeframe` (from `timeframe_setup`), `timeframes_supported` (context + setup + trigger, deduplicated)
- `family`, `regimes_supported`, `directions_supported`, `sessions_supported`, `providers_tested`, `version`, `description` (the idea's `synthesis`)
- `parameters: { initial_research_parameters: [...] }` — **nested under this exact key** so it can never be mistaken for tuned or locked values; `locked_parameters` stays empty until real research fixes something
- `limitations`/`risks`/`invalidation_conditions` — populated from `falsifiability.possible_biases`/`cost_sensitivity_risks`/`rejection_conditions` respectively
- `reports.ideas: [reportPath]` — the generation report is the only evidence behind this entry; there is no `reports.backtests`/`reports.validations` entry yet
- `notes` — a one-line provenance statement: *"Generated by tradingview-strategy-generator on \<date\>; not backtested; hypothesis: \<synthesis\>"*

## Hard rules

- Never write an idea whose quality gates didn't all pass (`accepted: true` from `runQualityGates` — see [quality-gates.md](quality-gates.md))
- Never write an idea the human operator didn't explicitly approve, even if every gate passed
- Never set `status` to anything other than `experimental` — this skill has no path to `validated`, `validation_pending`, or any other status
- Never invent a `pine_script_id` — it stays absent until a real Pine script exists and `pine_list_scripts` (in a future `tradingview-strategy-research` session) confirms its real ID
- Never invent `metrics` — they stay absent until a real `data_get_strategy_results` read exists
- `commitApprovedIdeas` can fold multiple approved entries into one registry write in a single pass — it never issues one `writeRegistry` call per idea
